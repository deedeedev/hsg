# hsg v1.0 — Publishable Release Plan

A staged plan for turning the personal `heisig_tatoeba` toolkit into a public,
audience-ready GitHub project (`hsg`).

Status: Draft
Last updated: 2026-07-01

---

## Motivation

The repository contains solid domain logic over local corpora (Heisig frames,
CC-CEDICT, Tatoeba, HSK, Renminwang, SUBTLEX-CH), plus a useful Anki <> Pleco
pipeline. In its current state it cannot reach a wide audience because:

- The name and module layout are tied to Heisig and Tatoeba specifically.
- Anki must be running locally for `stories` and the Pleco builder to work.
- Two CLI entry points (`hsg`, `hsg-tatoeba`, `hsg-cc`, `hsg-freq`) are split and inconsistent.
- Dead/stub code and a broken module land first in any visitor's reading.
- No README, no LICENSE, no tests, no CI, no published package.
- The "known characters" concept is hardcoded to Heisig, but the same logic
  applies to HSK learners, custom-list learners, and Anki/Pleco SRS users.

This document captures the four open strategic decisions and the six phases
of work needed to ship v1.0 on PyPI with a polished GitHub presence.

---

## Open strategic decisions

These four choices shape the plan. Unanswered, the phases below assume the
recommended default.

| # | Decision | Recommended default | Alternative |
|---|---|---|---|
| S1 | Project name / scope | Rename to a Hanzi-agnostic name (`hsg` retained as command; package `hzg` or `hanzi-coverage`) | Keep `heisig_tatoeba` and add other backends alongside |
| S2 | Anki / Pleco dependency | Drop AnkiConnect from required runtime; ship as optional plugin and one-shot importer | Keep AnkiConnect as a hard runtime dep |
| S3 | Language of comments / keywords | English-only code and docs; Italian becomes one of N i18n extensions | Keep Italian throughout, add EN layer |
| S4 | License and CLI breakage | Apache-2.0; allow breaking changes for v1.0 (no installed user base yet) | MIT; strict backward-compat with the four existing commands |

## Phases

### Phase 0 — Hygiene (1-2 days)

Quick wins that make the repo trustworthy on first visit. No feature work.

- Delete dead / broken code:
  - `utils/statistics.py` (imports non-existent `HeisigTools`)
  - `pysimpleguitable.py` (sandbox script)
  - `hsg gui` stub in `heisigtools.py`
  - unused imports (`sys` in `frequencytools.py`, `json` in `writers.py`)
  - broken `JsonWriter.writerow` (calls `json.dumps` without printing)
- Fix `setup.py` typo (`versio=`) and remove the duplicate Pipfile entries
  (`ht` + `hsg`).
- Remove unused `from ast import Try` (heisigtools.py:1).
- Migrate packaging to a single PEP 621 `pyproject.toml`
  (build backend `hatchling` or `setuptools`).
- Pin Python to >= 3.10; drop the Pipfile/Pipfile.lock in favour of
  `uv`-managed environment and a committed `uv.lock`.
- Consolidate the four entry points into one `cli` group with subcommands
  (matches the existing TODO at `heisigtools.py:21`):
  - `hsg parse` `hsg enrich` `hsg list` `hsg stories`
  - `hsg sentences` `hsg random` (formerly `hsg-tatoeba`)
  - `hsg lookup` (formerly `hsg-cc`)
  - `hsg freq` (formerly `hsg-freq`)
- Add `__main__.py` so `python -m hsg` works.

**Exit criteria:** repo installs cleanly via `uv pip install -e .`;
`hsg --help` lists all subcommands; no dead module imports.

---

### Phase 1 — Quality bar (2-3 days)

Make the repo visibly engineered. This is what a contributor judges before
opening a PR.

- **Ruff** for lint + format; commit `ruff.toml` config.
- **mypy** in `strict` mode via `pyproject.toml`; fix all current gaps.
- **pytest** scaffold with `click.testing.CliRunner`:
  - Unit tests for each `classes/*` module (Heisig, Ccedict, HSK,
    RenMinWang, SubtlexCh).
  - Command-level tests for every `hsg` subcommand against a small bundled
    fixture corpus.
  - Snapshot tests for `csv` / `json` / `tabulate` output.
- **pre-commit** hooks: ruff, ruff-format, mypy, end-of-file-fixer.
- **Logging** instead of `print` for diagnostics; keep `rich` only for
  user-facing output.
- **CI**: GitHub Actions matrix (Linux, macOS, Windows) x (Python 3.10, 3.11,
  3.12, 3.13). Jobs: `ruff`, `mypy`, `pytest --cov`, `build`.
- Coverage gate at 70% on PRs; target 85% before v1.0.

**Exit criteria:** all CI jobs green on `main`; coverage badge on README.

---

### Phase 2 — Decouple from Heisig (3-5 days)

The key change for reaching a wide audience. The killer feature is
"given a text, what % can I read + sentences I can read"; Heisig frame
numbers are just one proxy for the known set.

- Introduce a `KnownSet` interface (ABC) in `classes/knownset.py` with
  backends:
  - `heisig` (current behaviour, frame range)
  - `hsk` (`hsk_old` / `hsk_new` level cap)
  - `file` (one hanzi per line, a user-supplied list)
  - `anki-export` (parse Anki deck export locally, no AnkiConnect)
  - `pleco-flashcards` (Pleco history export)
- Refactor `Heisig` so it implements `KnownSet`; `heisigtools.parse` /
  `enrich` accept a `--known-set` and `--max` selector instead of hardcoded
  Heisig + `--max-frame`.
- Formalise the existing `Frequency` ABC (`classes/frequency.py`) and add
  a `CCEDICT` rank backend as a third corpus option.
- Pluggable story source: `hsg stories` reads from disk (CSV/JSON).
  Anki becomes a one-shot importer: `hsg stories import --from-anki
  --out stories.json`.
- Resurrect `tatoebasqlite.py` behind a `SentenceCorpus` interface; add a
  CC-CEDICT examples adapter as a second corpus.

**Exit criteria:** a learner who has never touched Heisig can install `hsg`,
point it at an Anki export, and run `hsg parse "..."` against their known
set. All existing Heisig workflows still pass their tests.

### Phase 3 — Polish (2-3 days)

What a GitHub visitor sees first.

- **README.md** with:
  - One-sentence problem statement ("given a Chinese text, what % can you
    read and which characters don't you know yet")
  - 30-second quickstart (`uvx hsg parse "..."`)
  - Feature tour with asciinema and screenshots
  - Comparison vs. Pleco reader, Chinese Text Analyser, Zhongwen extension,
    Languagecrush
  - Install (pipx / brew / pip), Usage, Configuration, Contributing, License
  - CI and coverage badges
- **LICENSE** (Apache-2.0 per S4).
- **CHANGELOG.md** generated by `release-please` (Conventional Commits).
- **CONTRIBUTING.md**, **CODE_OF_CONDUCT.md**, **SECURITY.md**.
- `.github/`: issue templates (bug, feature, corpus request), PR template,
  stale bot config.
- Docstrings on every public class/function; publish API reference via
  `mkdocs` + `mkdocstrings` (Material theme) on GitHub Pages.
- **CITATION.cff** for the "Cite this repository" button.
- Snapshot screenshots and an asciinema cast committed under `docs/`.

**Exit criteria:** README renders cleanly on GitHub; docs site builds and
deploys; LICENSE and CITATION.cff are present.

---

### Phase 4 — Distribution (1-2 days)

- PyPI release via trusted publishing from GitHub Actions.
- `uvx hsg ...` works out-of-the-box with no local install.
- Tag releases using semver; v1.0.0 marks the release described here.
- Optional extras: Homebrew tap, AUR package, Nix flake.
- GitHub Releases with sdist + pure-Python wheel for each platform.
- Optional binary wheel build via `cibuildwheel` only if Phase 2 adds a
  compiled dep (not currently needed).

**Exit criteria:** `pip install hsg` or `uvx hsg` produces a working CLI on
a clean machine; release artifacts are attached to a GitHub Release.

---

### Phase 5 — Growth features (ongoing)

Features that drive adoption once v1.0 is out.

- **Anki deck from unknown chars export**: `hsg parse` gains `--output
  anki` to emit a TSV import file (or generated `.apkg`) with hanzi +
  pinyin + sentence + audio hooks.
- **HTML report output**: single-file HTML with click-to-expand stories,
  frequency bars, colour-coded coverage. Shareable artifact.
- **Corpus builder**: `hsg corpus build` pulls Renminwang, SUBTLEX-CH,
  CC-CEDICT, Tatoeba, HSK, Heisig from upstream into a cache directory;
  document the data licences. Stop committing large binaries into the
  repo.
- **i18n extensions**: `assets/extensions/{lang}.tsv` with a loader; EN is
  default, IT preserved.
- **Profile-based config**: `~/.config/hsg/config.toml` storing the user's
  known-set backend, frequency corpus, language, default Anki deck; shorter
  CLI invocations.
- **Optional TUI** (`textual`) and **web UI** (`fastapi` + static page) —
  explicitly out of scope for v1.0; revisit after adoption.

## Acceptance criteria for v1.0

- One `hsg` command with all subcommands; no split entry points.
- A non-Heisig learner can install `hsg`, point it at an Anki export, and
  get text-coverage analysis and comprehensible Tatoeba sentences.
- `ruff`, `mypy --strict`, `pytest --cov` all green on Linux, macOS,
  Windows across Python 3.10-3.13.
- README, LICENSE, CONTRIBUTING, CHANGELOG, CITATION.cff, issue templates
  all present.
- PyPI release via trusted publishing; `uvx hsg --help` works on a clean
  machine.
- No hard dependency on Anki Connect or Pleco running locally.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Renaming breaks the user's muscle memory | Keep `hsg` as the command name; only the package/repo name changes |
| Bundled corpora have unclear licences | Phase 5 corpus builder fetches upstream and documents licences; remove large binaries from git history with `git filter-repo` before v1.0 if needed |
| Tatoeba TSV is from 2021 | Phase 5 corpus builder refreshes it from the official Tatoeba downloads |
| BACKWARD-compat breakage angers existing users | The "existing users" set is approximately the author; safe to break for v1.0 |
| Anki import path is complex | Treat as optional: ship the file-based stories source first, Anki importer comes in Phase 5 |
